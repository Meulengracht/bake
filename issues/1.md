### Enhancing Local Package Installation with Developer Proof and Overrides

To enable developers to install their own packages locally in the Chef Runtime Daemon (`serve`), we propose enhancements to address the following challenges:

1. **Proof Generation**: Local packages currently lack a cryptographically signed checksum (`proof`) when built. Proofs are only generated by the store during upload, using a private key per publisher.
2. **Revision Assignment**: Locally built packages are not assigned a revision, as this is also handled by the store. Revisions are unique identifiers per package version.

### Proposed Solution

#### 1. Generate a Developer Proof for Locally Built Packages
- When building a package locally using `bake`, automatically generate a `dev-proof` using the developer's private key to sign the built artifact.
- Developers will be required to set up their private key prior to building. The existing `order` tool, used for setting up Chef accounts, can be reused to automate private key generation and setup.
- Ensure the locally generated `dev-proof` is securely attached to the package and differentiated from store-generated proofs with a clear origin marker.

#### 2. Allow Local Package Installation Without Proof
- Introduce a CLI override switch in the `serve` tool to permit installations of packages without proof.
- Add safeguards to avoid unintentional use in environments where it is not suitable:
  - Display clear warnings when the override is used.
  - Optionally restrict the override by environment (e.g., allow only in development mode).

#### Other Considerations
- Telemetry: Log all package installations (with or without proof) to enable auditing and debugging.
- Key Management: Provide guidance for developers to secure their private keys. Optionally, implement key rotation support in future iterations.
- Usability: Ensure developer onboarding includes streamlined instructions to set up private keys, build, and install packages locally.

#### Benefits
- Improves security and authenticity of locally built packages.
- Provides flexibility for developers without compromising production integrity.
- Builds on existing infrastructure (`order` tool) for private key automation, ensuring consistency and ease of use.
